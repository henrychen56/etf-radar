# 主動式ETF雷達 — Netlify 自動抓取部署說明

## 這套件包含什麼：

```
netlify-etf/
├── index.html                              ← 網站本體
├── netlify.toml                            ← Netlify 設定檔
├── package.json                            ← 依賴套件清單
└── netlify/functions/
    ├── parser.js                           ← HTML 解析邏輯（MoneyDJ、群益、台新共3組）
    ├── fetch-etf-holdings.js               ← 排程函式：每天自動抓取10檔ETF（多來源fallback）
    └── get-etf-holdings.js                 ← 一般函式：給前端讀取最新資料
```

## ★ 多來源策略（本版重點更新）

10檔ETF不再只靠MoneyDJ單一來源，改為依優先序逐一嘗試，第一個成功就採用：

| ETF | 優先順序 |
|---|---|
| 00992A | ① 群益投信官網(detail/**500**/portfolio) → ② MoneyDJ |
| 00982A | ① 群益投信官網(detail/**399**/portfolio) → ② MoneyDJ |
| 00987A | ① 台新投信官網(ETFSeriesDetail/00987A) → ② MoneyDJ |
| 其餘7檔 | 僅 MoneyDJ（其餘投信官網經查證為禁止爬蟲或純JS頁面，無法抓取） |

⚠️ **重要修正記錄**：00992A與00982A在群益投信官網的product ID，先前對話中曾誤植為共用同一組ID(500)，這是錯的——已交叉比對多筆搜尋結果確認 **00992A=500、00982A=399** 是兩個不同ID，程式碼已修正。若群益投信網站改版調整ID，這裡需要重新查證更新。

## 部署步驟

1. **上傳整個資料夾**到 Netlify（拖拉整個 `netlify-etf` 資料夾到 Netlify 部署頁面，或用 Git 串接）
2. Netlify 偵測到 `netlify/functions` 資料夾，會自動部署這三個 Function
3. `fetch-etf-holdings.js` 內用 `schedule('30 0,5 * * *', handler)` 宣告了排程，Netlify 會自動辨識，**不需要額外在後台設定 cron**
4. 部署完成後，這個排程**要等到下一個排定時間點（台灣時間08:30或13:30，看部署當下哪個先到）才會第一次自動執行**——不會在部署當下立刻跑

## 排程時間說明

`30 0,5 * * *` 是 cron 語法，對應**台灣時間每天 08:30 與 13:30 各執行一次**（共兩次／天）：
- UTC 00:30 = 台灣 08:30（開盤前，抓前一交易日收盤後的最新公告）
- UTC 05:30 = 台灣 13:30（盤中，捕捉投信若有盤中更新的情況）

如果想改時間或頻率，修改 `fetch-etf-holdings.js` 最後一行的 cron 字串。例如：
- 改回一天一次：`'0 1 * * *'`（台灣早上9點）
- 改成一天三次：`'0 0,4,8 * * *'`（小時欄位用逗號分隔多個值即可）

⚠️ 不建議改成每小時一次。MoneyDJ 的持股資料本身是投信每天公告一次（不是每小時變動），抓更頻繁不會讓資料更準確，反而會：
- 增加被對方網站判定為異常流量、進而封鎖的風險
- 多倍消耗 Netlify Function 的免費執行額度

## 重要：手動觸發第一次測試

排程預設要等到隔天才會跑。若想**現在就測試**抓取功能是否正常：

- Netlify 的 Function 後台（Functions 頁籤）通常有「手動觸發」或「Test」按鈕，可以立即執行一次 `fetch-etf-holdings`，不用等排程時間
- 或者直接用瀏覽器打開 `https://你的網站.netlify.app/.netlify/functions/fetch-etf-holdings`（部分 Netlify 版本允許直接造訪 URL 觸發 Scheduled Function 進行測試，視帳號方案而定）

## 老實話：這套自動化系統可能遇到的問題

在把這份說明交給您之前，有幾個我**沒辦法在這個對話環境裡實際測試**、必須請您部署後親自驗證的風險點：

### 1. 三組解析器都是依「曾經抓到的單次內容」回推寫的，不是官方文件
`parser.js` 裡 MoneyDJ、群益投信官網、台新投信官網的 HTML 解析規則，都是依照之前用 `web_fetch` 觀察到的內容**回推猜測**寫的，不是真正取得官方API文件、或大量比對多次真實HTML逐行核對寫的。尤其**群益投信官網的解析器**最不確定——它的頁面不是傳統 `<table>` 結構，是用一連串重複區塊呈現資料，我用「轉純文字後逐行掃描」的方式硬解析，這種寫法对頁面排版的細微變化特別敏感，改版風險較高。

**怎麼確認**：部署後第一次執行完，去 Netlify 後台看 Function 的 log，裡面會印出每檔ETF「嘗試了哪些來源、各自成功或失敗、失敗原因代碼」，例如：
```
00992A←群益投信官網, 00982A←群益投信官網, 00987A←台新投信官網, 00981A←MoneyDJ理財網...
```
若群益或台新失敗，會自動退回MoneyDJ，您看log就知道是退回了還是兩個都失敗。

### 2. 之前實測過的「快取串檔」問題，伺服器端不保證不會發生
我們在這個對話前段，用 `web_fetch` 測試時發現 MoneyDJ 偶爾會回傳錯誤的ETF頁面內容（要00992A卻拿到0050）。程式裡的 `verifyCorrectETF()` 防呆機制只套用在 MoneyDJ 這個來源（核對頁面標題裡的代號），群益、台新官網因為是「不同URL對應不同ETF」（不是同一URL換參數），串檔風險的性質不同，所以**沒有對這兩個來源做同樣的title驗證**——如果您發現群益/台新抓到的資料代號對不上，請回報，我再補上對應的驗證機制。

### 3. 投信官網可能比MoneyDJ更容易被反爬蟲機制擋下
群益、台新官網過去沒有被大量自動化請求過，不確定是否有針對性的反爬蟲規則（例如IP頻率限制、需要特定Cookie等）。如果這兩個來源持續失敗，系統會自動退回MoneyDJ，不會讓整個功能掛掉，但代表「較即時」這個優勢就拿不到了。

### 4. MoneyDJ 可能會擋自動化請求
我在請求加了瀏覽器 User-Agent 偽裝，但如果 MoneyDJ 有更嚴格的反爬蟲機制（例如要求 JavaScript 執行、驗證碼、或封鎖雲端伺服器IP段），這個簡單的 `fetch()` 可能直接被拒絕或拿到不完整內容。

### 5. Netlify Blobs 是否可用，依您的方案而定
`@netlify/blobs` 在免費方案（Starter）通常可用，但有用量限制。如果您的帳號版本較舊或方案不同，可能需要到 Netlify 後台確認 Blobs 功能已啟用。

## 如果自動抓取失敗，網站還能用嗎？

**能。** 這是設計上特別保留的後路：

- 網站開啟時，會先用我之前手動驗證寫死在 `index.html` 裡的 **4檔真實資料**（00992A、00982A、00994A、00981A）墊底顯示
- 接著才非同步去呼叫 Netlify Function，如果失敗（排程還沒跑、Function 部署有問題、MoneyDJ 擋掉等任何原因），網站會**安靜地維持原本的4檔資料，不會整個壞掉或顯示空白**
- 「📋 貼上更新」功能完全保留，您隨時可以手動補資料，跟自動抓取並存

## 建議的驗收流程

1. 部署上 Netlify
2. 去 Function 後台手動觸發一次 `fetch-etf-holdings`
3. 看執行 log，確認「成功幾檔／失敗幾檔」，以及 **00992A、00982A、00987A 這三檔實際用的是官網還是退回MoneyDJ**
4. 打開網站，看右上角資料狀態列的「自動抓取」文字，確認是否顯示「🟢 自動抓取：X檔資料」
5. 點開 00992A 或 00982A 的「ETF持股」分頁，核對裡面的持股權重數字，跟您自己去群益官網看到的是否一致（這是最直接判斷解析器有沒有抓對的方法）
6. 若失敗檔數較多，把 log 裡的錯誤訊息（如 `PARSE_EMPTY`、`WRONG_ETF_RETURNED`、`HTTP_403`、`ALL_SOURCES_FAILED`）回報給我，我再依實際錯誤原因調整 `parser.js` 的解析規則
